## 文档说明

本规范是 LifeVerse 产品所有界面与交互的统一设计语言基线，覆盖设计理念、色彩、字体、间距、组件、动效、交互、响应式、无障碍与信息架构十大维度。所有新页面、新组件、新动效必须以本规范为唯一视觉与交互准绳；已有实现如与本规范冲突，以本规范为准并排期对齐。

本规范与各组件级 `spec.md` 的关系：本规范定义"全局设计语言"，组件 spec 定义"单组件实现细节"。当组件 spec 未覆盖时，回退到本规范。

## 1. 设计理念

### 1.1 产品调性

LifeVerse 是一个 AI 生命操作系统，它的视觉与交互必须承载"生命"的重量感与"宇宙"的辽阔感。我们用四个关键词锚定整体调性：

| 调性关键词 | 视觉翻译 | 交互翻译 | 反例（要避免） |
| --- | --- | --- | --- |
| 深邃 | 深黑底色、留白、星点粒子 | 信息分层揭示，不一次倾倒 | 亮色背景、信息堆砌 |
| 温暖 | 金色点缀、暖色情感色调 | 陪伴式文案、先陪伴后揭示 | 冷冰冰的数据罗列 |
| 智慧 | 衬线标题、克制装饰 | 追问式交互、多元视角并存 | 花哨动效、单一结论 |
| 仪式感 | 居中构图、入场动画、光晕 | 议会仪式流程、报告生成仪式 | 即时弹窗、无过渡硬切 |

这四个调性不是并列的，而是有优先级：**深邃是底色，温暖是温度，智慧是骨架，仪式感是节奏**。当四者冲突时，按此顺序取舍。

### 1.2 设计原则

LifeVerse 的设计原则直接继承自世界观层的三条哲学，并将其翻译为可执行的设计判断标准。

#### 原则一：多元优先于正确

- 界面永远呈现多个视角，而非一个"标准答案"。
- 议会大厅中 7 位智者的发言并列展示，不折叠、不排序为"最佳"。
- 命运报告保留分歧段落，而非抹平为单一结论。
- 价值雷达支持多人格叠加显示，让差异可见。
- **设计判断**：当需要在"清晰单一"与"多元并存"之间选择时，选择并存。

#### 原则二：冲突优先于和谐

- 冲突是产品价值，不是 bug。冲突可视化必须显性、可感知。
- 议会中智者之间的冲突用红色连线 + 脉冲动画呈现，不弱化。
- 价值雷达 compare 模式让两个对立人格左右镜像对比。
- 内心世界的人格冲突用粒子震荡效果表达。
- **设计判断**：不要用"和谐"的视觉语言掩盖冲突，冲突越剧烈，视觉越要诚实。

#### 原则三：追问优先于结论

- 交互流程以"提问"驱动，而非"给答案"。
- 苏格拉底式追问在 UI 中以打字机效果逐字呈现，制造思考节奏。
- 报告结尾永远以"留给你思考的问题"收束，而非"建议你这样做"。
- 空状态用引导性提问代替功能说明。
- **设计判断**：每个界面都要问自己——这里是在给结论，还是在引发追问？优先后者。

### 1.3 情感设计目标

LifeVerse 的用户多处于人生转折、迷茫、回望的状态，情感设计是产品的核心竞争力，而非锦上添花。我们设定三个递进的情感目标：

| 层级 | 目标 | 用户感受 | 设计手段 |
| --- | --- | --- | --- |
| L1 | 被理解 | "它懂我现在的处境" | 个性化画像、记忆回放、情绪扫描 |
| L2 | 被陪伴 | "它和我一起走过这段" | 议会仪式、AI 亲人、温柔文案、粒子陪伴 |
| L3 | 被启发 | "它让我看见了新的可能" | 多元视角、时间推演、价值雷达漂移 |

**情感设计红线**：

1. 永远不制造焦虑。当数据残酷时（如高后悔概率），先提供 Reunion 入口，再揭示数据。
2. 永远不替代决策。所有报告结尾必须有"最终的选择权在你"。
3. 永远不冷漠。错误提示、空状态、加载状态都要有温度，禁止出现"出错了""无数据"等冰冷文案。
4. 检测到自伤倾向时，主动提供专业心理热线，这是最高优先级的情感安全机制。

### 1.4 设计语言总纲

将上述理念收敛为可执行的设计语言总纲，作为所有视觉决策的元规则：

- **底色**：深黑为主，金色为魂，暖冷中性三色调随情感流转。
- **构图**：居中对称为主（仪式感），非对称为辅（动态感）。
- **节奏**：入场有仪式，过渡有呼吸，反馈有触感。
- **信息**：分层揭示，先陪伴后揭示，先追问后结论。
- **材质**：毛玻璃 + 光晕 + 粒子，营造"宇宙"质感，避免扁平化。

## 2. 色彩系统

### 2.1 色彩总览

LifeVerse 采用深色主题为主，金色为唯一品牌强调色。色彩系统分为五个层级：背景层、品牌层、文字层、功能层、情感层。所有颜色值已对齐 `tailwind.config.ts` 中的 token，前端必须使用 Tailwind token 而非硬编码色值。

### 2.2 背景层

背景层定义页面的纵深空间，从最深的宇宙底色到卡片表面，形成 4 级纵深。

| Token | 色值 | 用途 | Tailwind 类 |
| --- | --- | --- | --- |
| `bg.DEFAULT` | `#060710` | 页面主背景，宇宙底色 | `bg-bg` |
| `bg.soft` | `#0c0e1a` | 次级背景，分区背景 | `bg-bg-soft` |
| `bg.card` | `#0d0f1a` | 卡片背景 | `bg-bg-card` |
| `bg.card-hover` | `#1a1d2e` | 卡片悬停背景 | `bg-bg-card-hover` |

> 注：本规范将卡片背景统一为 `#0d0f1a`（对应 `bg.card`），与 `tailwind.config.ts` 中 `#12141f` 略有差异，以本规范为准，需同步更新 tailwind 配置。

纵深使用规则：
- 页面最底层使用 `bg.DEFAULT` + 粒子背景。
- 大分区（如侧栏、底部栏）使用 `bg.soft`。
- 内容卡片使用 `bg.card`，hover 时过渡到 `bg.card-hover`。
- 弹窗、抽屉使用 `bg.card` + 毛玻璃滤镜。

### 2.3 品牌层（金色系）

金色是 LifeVerse 的唯一品牌色，象征智慧与仪式感。金色系包含 3 个梯度，用于不同强度的强调。

| Token | 色值 | 用途 | Tailwind 类 |
| --- | --- | --- | --- |
| `gold.DEFAULT` | `#c9a84c` | 主品牌色，按钮、焦点、高亮 | `text-gold` / `bg-gold` / `border-gold` |
| `gold.soft` | `#c9a84c33` | 半透明金，光晕、背景填充 | `bg-gold-soft` |
| `gold.dim` | `#8a7430` | 暗金，次要强调、禁用态 | `text-gold-dim` |

金色使用规则：
- 金色是稀缺资源，每屏不超过 3 处强金色，避免视觉通胀。
- 金色用于：主 CTA 按钮、焦点边框、关键数据高亮、仪式光晕。
- 金色不用于：正文文字、大面积背景、普通边框。
- 金色与深黑的对比度为 7.2:1，满足 AAA 级别。

### 2.4 文字层

文字层定义信息的层级关系，通过明度差异建立阅读优先级。

| Token | 色值 | 用途 | 对比度（对 #060710） | Tailwind 类 |
| --- | --- | --- | --- | --- |
| `text.primary` | `#e8e6e3` | 主文字，标题、正文 | 15.8:1 | `text-[#e8e6e3]` |
| `text.secondary` | `#8a8892` | 次要文字，说明、时间戳 | 6.9:1 | `text-[#8a8892]` |
| `text.tertiary` | `#555560` | 微弱文字，占位、禁用 | 2.8:1 | `text-[#555560]` |

> 注：`text.tertiary` 对比度低于 4.5:1，仅用于非关键装饰性文字（如占位符、分割线标签），不可用于承载信息的文字。

文字层级使用规则：
- 一屏之内必须有且只有一个最优先的文字层级（primary），避免视觉平铺。
- secondary 用于辅助说明、元信息（时间、作者、标签）。
- tertiary 仅用于占位符、水印、装饰性序号，不承载语义。
- 金色文字（`text-gold`）用于极少数关键强调，如报告标题、仪式文案。

### 2.5 功能色

功能色用于状态反馈，每个功能色有明确的语义边界，不可混用。

| 语义 | Token | 色值 | 用途 | Tailwind 类 |
| --- | --- | --- | --- | --- |
| 成功 | `success` | `#4ade80` | 操作成功、共识达成、稳定维度 | `text-[#4ade80]` |
| 警告 | `warning` | `#f87171` | 错误、冲突、风险、后悔概率高 | `text-[#f87171]` |
| 信息 | `info` | `#60a5fa` | 提示、链接、自由维度、冷色调 | `text-[#60a5fa]` |

> 注：本规范采用 `#4ade80` / `#f87171` / `#60a5fa` 作为功能色标准值，与 `tailwind.config.ts` 中 `#5de8a0` / `#e85d5d` / `#5da0e8` 略有差异，以本规范为准，需同步更新 tailwind 配置以统一色板。

功能色使用规则：
- 成功色用于：保存成功、议会共识、价值雷达稳定维度高亮。
- 警告色用于：操作失败、议会冲突连线、后悔概率 ≥ 50% 警示、删除二次确认。
- 信息色用于：中性提示、可点击链接、价值雷达自由维度高亮。
- 功能色不用于大面积填充，仅用于图标、边框、小面积高亮。

### 2.6 边框与分割

| Token | 色值 | 用途 | Tailwind 类 |
| --- | --- | --- | --- |
| `border.gold` | `rgba(201, 168, 76, 0.15)` | 金色边框，卡片、焦点区域 | `border-gold-soft` |
| `border.DEFAULT` | `#1e2235` | 默认边框，分割线、卡片边框 | `border-border` |
| `border.soft` | `#161929` | 次级边框，内部分割 | `border-border-soft` |

边框使用规则：
- 卡片默认边框为 `border.DEFAULT`，hover 时过渡为 `border.gold`（金色微光边框）。
- 焦点状态边框为 `border.gold` + 金色光晕（box-shadow）。
- 分割线使用 `border.soft`，1px，不抢视觉。

### 2.7 情感色调

情感色调是 LifeVerse 独有的色彩维度，用于标记记忆、情绪、人格的情感属性。情感色调不是固定色值，而是色温倾向，会以半透明叠加方式影响元素的光照。

| 情感色调 | 色相 | 代表色 | 语义 | 应用场景 |
| --- | --- | --- | --- | --- |
| 暖 | 金色系 | `#c9a84c` | 温暖、幸福、爱、骄傲 | 爱情海洋、家庭小镇、幸福维度 |
| 冷 | 蓝色系 | `#60a5fa` | 清冷、理性、遗憾、思念 | 成长山脉、自由维度、冷色调记忆 |
| 中性 | 灰色系 | `#8a8892` | 平静、客观、过渡 | 日常记忆、系统消息、中性状态 |

情感色调应用规则：
- 记忆星球中每条记忆的"光照"由其情感色调决定，暖色记忆发金光，冷色记忆发蓝光。
- 内心世界的内心天气用情感色调表达当日情绪基调。
- 价值雷达的 5 个维度各有专属色：自由=蓝、财富=金、幸福=红、稳定=绿、成长=紫。
- 情感色调以 `opacity 0.15-0.3` 叠加，不覆盖基础色。

### 2.8 雷达图维度配色

价值雷达的 5 个维度有固定的专属配色，跨模块统一，不可变更。

| 维度 | 色值 | 语义 | Tailwind 类 |
| --- | --- | --- | --- |
| 自由 freedom | `#3498DB` 蓝 | 自主、不受束缚 | `text-[#3498DB]` |
| 财富 wealth | `#F1C40F` 金 | 物质、财务安全 | `text-[#F1C40F]` |
| 幸福 happiness | `#E74C3C` 红 | 幸福感、关系质量 | `text-[#E74C3C]` |
| 稳定 stability | `#2ECC71` 绿 | 稳定、安全感 | `text-[#2ECC71]` |
| 成长 growth | `#9B59B6` 紫 | 成长、自我超越 | `text-[#9B59B6]` |

### 2.9 色彩使用禁忌

1. 禁止使用纯白（`#ffffff`）作为文字色，使用 `#e8e6e3` 代替，避免深色背景下过于刺眼。
2. 禁止使用纯黑（`#000000`）作为背景色，使用 `#060710` 代替，保留宇宙的微光感。
3. 禁止在大面积区域使用高饱和度色，高饱和色仅用于小面积强调。
4. 禁止将功能色用于装饰，功能色必须承载语义。
5. 禁止在同一界面使用超过 3 种情感色调，避免色彩混乱。

### 2.10 暗色模式与高对比度模式

LifeVerse 默认为深色主题，不提供浅色主题切换（深色是产品调性的核心组成部分）。但提供高对比度模式以满足无障碍需求：

- 高对比度模式：背景加深为纯黑 `#000000`，文字提亮为纯白 `#ffffff`，金色加深为 `#ffd700`，边框加粗为 2px。
- 触发方式：跟随系统 `prefers-contrast: more` 自动切换，或用户在设置中手动开启。

## 3. 字体系统

### 3.1 字体家族

LifeVerse 采用双字体策略：衬线体承载仪式感与智慧感，无衬线体承载可读性与现代感。

| 用途 | 字体 | 备选 | 引入方式 | Tailwind 类 |
| --- | --- | --- | --- | --- |
| 标题 | Instrument Serif | Georgia, serif | `next/font/google` | `font-serif` |
| 正文 | Work Sans | -apple-system, BlinkMacSystemFont, sans-serif | `next/font/google` | `font-sans` |
| 代码 | SF Mono | Fira Code, Cascadia Code, Consolas, monospace | 系统字体 | `font-mono` |

字体加载策略：
- 使用 `next/font/google` 加载 Instrument Serif 与 Work Sans，实现字体自托管与自动优化。
- Instrument Serif 加载字重：400（Regular）、500（Medium）。
- Work Sans 加载字重：300（Light）、400（Regular）、500（Medium）、600（SemiBold）、700（Bold）。
- 设置 `display: swap`，避免字体加载阻塞渲染。
- 预加载关键字体子集（拉丁字母 + 中文常用字）。

### 3.2 字号层级

字号采用模块化比例（modular scale），基于 16px 基准字号，使用约 1.5 倍率递增。

| 层级 | Token | 桌面端 | 移动端 | 行高 | 字重 | 字体 | 用途 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| H1 | `text-h1` | 80px | 48px | 1.1 | 400 | Serif | 首页 Hero、报告主标题 |
| H2 | `text-h2` | 48px | 32px | 1.2 | 400 | Serif | 页面标题、模块标题 |
| H3 | `text-h3` | 24px | 20px | 1.3 | 500 | Serif | 区块标题、卡片标题 |
| H4 | `text-h4` | 20px | 18px | 1.4 | 500 | Sans | 子标题、侧栏标题 |
| Body | `text-body` | 16px | 16px | 1.6 | 400 | Sans | 正文、对话内容 |
| Body Small | `text-body-sm` | 14px | 14px | 1.6 | 400 | Sans | 辅助正文、列表项 |
| Caption | `text-caption` | 12px | 12px | 1.5 | 400 | Sans | 标签、时间戳、说明 |
| Overline | `text-overline` | 12px | 12px | 1.4 | 600 | Sans | 段首标签、分类标记 |

Tailwind 配置映射：

```typescript
// tailwind.config.ts 字号扩展
fontSize: {
  h1: ['clamp(48px, 5vw, 80px)', { lineHeight: '1.1', fontWeight: '400' }],
  h2: ['clamp(32px, 4vw, 48px)', { lineHeight: '1.2', fontWeight: '400' }],
  h3: ['clamp(20px, 2vw, 24px)', { lineHeight: '1.3', fontWeight: '500' }],
  h4: ['clamp(18px, 1.5vw, 20px)', { lineHeight: '1.4', fontWeight: '500' }],
  body: ['16px', { lineHeight: '1.6', fontWeight: '400' }],
  'body-sm': ['14px', { lineHeight: '1.6', fontWeight: '400' }],
  caption: ['12px', { lineHeight: '1.5', fontWeight: '400' }],
  overline: ['12px', { lineHeight: '1.4', fontWeight: '600', letterSpacing: '0.08em', textTransform: 'uppercase' }],
}
```

### 3.3 行高规范

行高直接影响阅读节奏，LifeVerse 根据文字用途设定差异化行高：

| 文字类型 | 行高 | 理由 |
| --- | --- | --- |
| 标题（H1-H3） | 1.1 - 1.3 | 标题字号大，紧凑行高避免松散 |
| 正文（Body） | 1.6 | 正文需要舒适阅读节奏，1.6 是中文阅读最佳行高 |
| 辅助文字（Caption） | 1.5 | 小字号需要略松行高保证可读 |
| 长文本（报告、对话） | 1.7 | 长篇阅读内容需要更松的行高减轻疲劳 |

### 3.4 字重规范

| 字重 | 数值 | 用途 | 字体 |
| --- | --- | --- | --- |
| Light | 300 | 大标题辅助、次要英文 | Work Sans |
| Regular | 400 | 正文、标题（Serif） | Instrument Serif / Work Sans |
| Medium | 500 | 子标题、强调正文 | Work Sans |
| SemiBold | 600 | 段首标签、按钮文字 | Work Sans |
| Bold | 700 | 极少使用，仅用于关键数据 | Work Sans |

字重使用规则：
- 衬线体（Instrument Serif）仅使用 400 与 500 两个字重，衬线体不适合过粗。
- 无衬线体（Work Sans）避免使用 Bold（700），用 SemiBold（600）代替以保持优雅。
- 通过字号与字色建立层级，而非过度依赖字重对比。

### 3.5 字间距规范

| 文字类型 | 字间距 | 说明 |
| --- | --- | --- |
| 中文标题 | 0 | 中文不需要额外字间距 |
| 英文标题 | -0.02em | 大字号英文略收紧，提升精致感 |
| 正文 | 0 | 默认字间距 |
| Overline 标签 | 0.08em | 全大写标签加宽字间距，提升识别度 |
| 数字数据 | -0.02em | 数字略收紧，避免松散 |

### 3.6 文字使用禁忌

1. 禁止正文使用衬线体，衬线体仅用于标题与仪式性文案。
2. 禁止标题使用无衬线体，标题的仪式感由衬线体承载。
3. 禁止使用下划线表示强调，使用字色或字重代替。
4. 禁止中英文混排时字号不一致，中英文必须使用同一字号层级。
5. 禁止正文单行超过 75 个中文字符，超过应换行或分栏。

## 4. 间距系统

### 4.1 基准单位

LifeVerse 的间距系统基于 **4px 基准单位**，所有间距必须是 4 的倍数。这保证了整个界面的节奏感与一致性。

```typescript
// tailwind.config.ts 间距扩展
spacing: {
  0: '0px',
  1: '4px',    // 基准单位
  2: '8px',
  3: '12px',
  4: '16px',
  5: '20px',
  6: '24px',
  8: '32px',
  10: '40px',
  12: '48px',
  16: '64px',
  24: '96px',
  32: '128px',
}
```

### 4.2 间距层级

| Token | 数值 | Tailwind 类 | 用途 |
| --- | --- | --- | --- |
| `space.xs` | 4px | `p-1` / `m-1` / `gap-1` | 图标与文字间距、极小间距 |
| `space.sm` | 8px | `p-2` / `m-2` / `gap-2` | 紧凑元素间距、标签内边距 |
| `space.md` | 12px | `p-3` / `m-3` / `gap-3` | 小组件间距、表单元素间距 |
| `space.base` | 16px | `p-4` / `m-4` / `gap-4` | 组件间距（紧凑）、列表项间距 |
| `space.lg` | 24px | `p-6` / `m-6` / `gap-6` | 组件间距（标准）、卡片内边距 |
| `space.xl` | 32px | `p-8` / `m-8` / `gap-8` | 组件间距（宽松）、区块间距 |
| `space.2xl` | 48px | `p-12` / `m-12` / `gap-12` | 区块间距、页面边距（桌面） |
| `space.3xl` | 64px | `p-16` / `m-16` / `gap-16` | 大区块间距、首屏分区 |
| `space.4xl` | 96px | `p-24` / `m-24` / `gap-24` | 页面级垂直间距、Hero 区域 |

### 4.3 页面边距

页面边距指内容区域与视口边缘的距离，根据设备类型差异化设定。

| 设备 | 边距 | Tailwind 类 | 说明 |
| --- | --- | --- | --- |
| 移动端（< 768px） | 24px | `px-6` | 保证内容不被边缘切割 |
| 平板（768-1024px） | 32px | `px-8` | 过渡边距 |
| 桌面（≥ 1024px） | 48px | `px-12` | 充足留白，聚焦内容 |
| 大屏（≥ 1440px） | 64px | `px-16` | 最大边距，内容居中 |

最大内容宽度限制：
- 文本内容页：`max-w-3xl`（768px），保证阅读舒适度。
- 卡片网格页：`max-w-7xl`（1280px），充分利用屏幕。
- 议会大厅：`max-w-full`，全屏沉浸式体验。
- 报告页：`max-w-4xl`（896px），兼顾阅读与图表。

### 4.4 卡片内边距

| 卡片类型 | 内边距 | Tailwind 类 | 说明 |
| --- | --- | --- | --- |
| 标准卡片 | 24px | `p-6` | 默认卡片内边距 |
| 紧凑卡片 | 16px | `p-4` | 列表项、小卡片 |
| 宽松卡片 | 32px | `p-8` | 报告卡片、重要内容卡片 |
| Hero 卡片 | 48px | `p-12` | 首页 Hero、仪式区域 |

### 4.5 组件间距

组件间距指同级组件之间的垂直间距，根据内容密度选择。

| 间距类型 | 数值 | Tailwind 类 | 适用场景 |
| --- | --- | --- | --- |
| 紧凑 | 16px | `gap-4` / `space-y-4` | 表单字段、列表项、标签组 |
| 标准 | 24px | `gap-6` / `space-y-6` | 卡片之间、区块内组件 |
| 宽松 | 32px | `gap-8` / `space-y-8` | 区块之间、大组件之间 |

### 4.6 圆角系统

| Token | 数值 | Tailwind 类 | 用途 |
| --- | --- | --- | --- |
| `radius.sm` | 8px | `rounded-sm` | 小按钮、标签、输入框 |
| `radius.DEFAULT` | 14px | `rounded-default` | 卡片、按钮、弹窗 |
| `radius.lg` | 20px | `rounded-2xl` | 大卡片、Hero 区域 |
| `radius.full` | 9999px | `rounded-full` | 头像、圆形按钮、胶囊标签 |

### 4.7 间距使用规则

1. 所有间距必须使用 Tailwind spacing token，禁止硬编码像素值。
2. 垂直方向使用 `space-y-*`，水平方向使用 `gap-*`，避免 margin 塌陷。
3. 嵌套元素的间距应从内到外递增，内层紧凑、外层宽松。
4. 同级元素的间距必须一致，不可出现"差不多但不完全相同"的间距。

## 5. 组件规范

本章定义 LifeVerse 全局通用组件的视觉与交互规范。各组件的详细实现规格见 `components/*/spec.md`，本章提供跨组件的统一标准。

### 5.1 按钮（Button）

按钮是 LifeVerse 中最高频的交互元素，定义 4 种变体与 3 种尺寸。

#### 5.1.1 变体

| 变体 | 视觉 | 用途 | Tailwind 类组合 |
| --- | --- | --- | --- |
| Primary | 金色背景 + 深色文字 | 主操作（提交、开启议会） | `bg-gold text-bg hover:bg-gold-dim` |
| Secondary | 透明背景 + 金色边框 + 金色文字 | 次操作（取消、返回） | `border border-gold text-gold hover:bg-gold-soft` |
| Ghost | 透明背景 + 次要文字色 | 辅助操作（筛选、排序） | `text-secondary hover:text-primary hover:bg-bg-card-hover` |
| Gold | 金色渐变背景 + 深色文字 + 光晕 | 仪式性操作（生成报告、开启仪式） | `bg-gradient-to-r from-gold-dim to-gold text-bg shadow-[0_0_24px_rgba(201,168,76,0.4)]` |

| 尺寸 | 高度 | 内边距 | 字号 | Tailwind 类 |
| --- | --- | --- | --- | --- |
| sm | 32px | `px-3 py-1.5` | 14px | `h-8 px-3 text-sm` |
| md | 40px | `px-5 py-2` | 16px | `h-10 px-5 text-base` |
| lg | 48px | `px-8 py-3` | 18px | `h-12 px-8 text-lg` |

#### 5.1.3 状态

| 状态 | 视觉变化 | 触发 |
| --- | --- | --- |
| Default | 基础样式 | 默认 |
| Hover | 背景加深 / 出现光晕 | 鼠标悬停 |
| Active | scale(0.95) 缩放 | 按下 |
| Focus | 金色边框 + 金色光晕 | 键盘聚焦 |
| Disabled | opacity 0.4 + cursor not-allowed | 禁用 |
| Loading | 文字替换为加载图标 + 旋转 | 异步操作中 |

#### 5.1.4 按钮规范要点

- 所有按钮圆角统一为 `rounded-default`（14px）。
- 按钮文字使用 Work Sans SemiBold（600），不使用全大写。
- 按钮内图标与文字间距 8px，图标在左。
- Primary 按钮每屏不超过 1 个，避免主操作竞争。
- Gold 按钮仅用于仪式性场景，不可滥用。

### 5.8 雷达图（Radar Chart）

雷达图是 LifeVerse 的核心可视化组件，详细实现见 `components/radar-chart/spec.md`，此处定义全局规范。

#### 5.8.1 基础规范

- 实现：纯 SVG，5 维正五边形。
- 维度：自由、财富、幸福、稳定、成长（顺时针，从顶部开始）。
- 网格：5 层同心五边形，颜色 `rgba(138, 136, 146, 0.15)`。
- 轴线：从中心到顶点，颜色 `rgba(138, 136, 146, 0.2)`。
- 数据填充：半透明填充（opacity 0.3）+ 描边（2px）。
- 描边动画：`stroke-dashoffset` 从周长到 0，duration 1.2s。
- 数值跳动：0 到目标值 CountUp，duration 0.8s，5 维依次跳动 stagger 100ms。

#### 5.8.2 尺寸规范

| 场景 | 尺寸 | 半径 |
| --- | --- | --- |
| Agent 卡片（small） | 80×80px | 30px |
| Agent 卡片（medium） | 120×120px | 45px |
| Agent 卡片（large） | 160×160px | 60px |
| 命运报告 | 280×280px | 110px |
| 全屏对比 | 400×400px | 160px |

#### 5.8.3 叠加模式

- `stack` 模式：多组数据半透明叠加，最多 5 组，透明度自动计算。
- `compare` 模式：两组数据左右镜像对比，中间显示差异值，用于冲突可视化。

### 5.9 组件通用规范

#### 5.9.1 过渡时长

所有组件的状态过渡统一使用以下时长：

| 过渡类型 | 时长 | 缓动 |
| --- | --- | --- |

## 6. 动效规范

### 6.1 动效设计哲学

LifeVerse 的动效不是装饰，而是情感的载体。每一段动效都必须回答一个问题：它让用户感受到了什么？动效设计遵循三条原则：

1. **有节奏**：入场有仪式，过渡有呼吸，反馈有触感。动效时长有明确的层级差异，不全部一样快，也不全部一样慢。
2. **有温度**：动效曲线以"先快后慢"为主（easeOutExpo），营造"温柔落地"的感觉，避免线性动效的机械感。
3. **有意义**：每个动效都承载信息——入场动效提示层级关系，反馈动效确认操作，仪式动效营造庄重感。无意义的动效一律删除。

### 6.2 全局缓动函数

LifeVerse 全局使用统一的缓动函数，所有动效必须使用以下缓动之一，禁止自定义。

| 缓动名称 | 贝塞尔曲线 | 适用场景 | Framer Motion 配置 |
| --- | --- | --- | --- |
| easeOutExpo | `[0.22, 1, 0.36, 1]` | 默认缓动，入场、过渡 | `ease: [0.22, 1, 0.36, 1]` |
| easeInOut | `[0.4, 0, 0.2, 1]` | 状态切换、双向过渡 | `ease: [0.4, 0, 0.2, 1]` |
| easeOutBack | `[0.34, 1.56, 0.64, 1]` | 弹性入场（卡片、图标） | `ease: [0.34, 1.56, 0.64, 1]` |
| linear | `linear` | 仅用于循环动画（旋转、脉冲） | `ease: "linear"` |

> `[0.22, 1, 0.36, 1]` 是 LifeVerse 的标志性缓动，所有主要动效默认使用此缓动。它的特征是"快速启动，温柔减速"，营造优雅的落地感。

### 6.3 页面过渡

页面切换使用 `fadeInUp` 过渡，营造"内容从下方浮起"的感觉。

```typescript
// Framer Motion 页面过渡变体
const pageVariants = {
  initial: {
    opacity: 0,
    y: 24,
  },
  animate: {
    opacity: 1,
    y: 0,

    },
  },
  exit: {
    opacity: 0,
    y: -12,
    transition: {
      duration: 0.3,
      ease: [0.4, 0, 0.2, 1],
    },
  },
};
```

| 属性 | 入场值 | 退场值 |
| --- | --- | --- |
| opacity | 0 → 1 | 1 → 0 |
| y | 24px → 0 | 0 → -12px |
| duration | 0.6s | 0.3s |
| ease | `[0.22, 1, 0.36, 1]` | `[0.4, 0, 0.2, 1]` |

卡片列表使用 stagger（错峰）入场，每张卡片间隔 0.1s 依次出现，营造"逐个亮相"的仪式感。

冲突可视化规范：

- 连线样式：实线，带电流脉冲动画（opacity + strokeWidth 循环）。
- 冲突解决后：红色连线渐变为金色（`#c9a84c`），持续 0.5s 后淡出。

| 时间轴 | 动效 | 说明 |
| --- | --- | --- |
| 0-0.5s | 粒子从四周向中心汇聚 | 营造"召集"感 |

| 0.5-1.0s | 中央议题区从中心展开 | scale 0 → 1 + opacity |
| 2.0-3.0s | Chairman 最后入场 + 金色光柱 | 仪式高潮 |
| 2.5-3.0s | 粒子背景恢复 ambient 模式 | 仪式结束 |

Agent 入场动画：
- 起点：中心点（scale 0，opacity 0）。
- 终点：各自位置（scale 1，opacity 1）。
- duration：0.6s，缓动 `[0.22, 1, 0.36, 1]`。
- stagger：200ms。

报告生成是议会的收束仪式，动效从"汇聚"到"展开"。

| 阶段 | 时长 | 动效 |
| --- | --- | --- |

| 展开 | 0.8s | 报告卡片各维度依次显现（stagger 100ms） |
| 粒子 | 1.0s | celebrate 模式粒子爆发 |

### 6.12 光晕动画

光晕是 LifeVerse 表达"生命感"的核心视觉语言。

| 场景 | 光晕颜色 | 周期 | 透明度范围 |
| --- | --- | --- | --- |
| Agent idle | 主题色 | 4s | 0.3 ↔ 0.5 |
| Agent speaking | 主题色 | 2s | 0.5 ↔ 1.0 |
| Agent conflict | `#f87171` | 0.8s | 0.4 ↔ 1.0 |
| 仪式光柱 | `#c9a84c` | 单次 | 0 → 0.8 |
| 报告汇聚 | `#c9a84c` | 单次 | 0.4 → 1.0 → 0 |

### 6.13 动效性能规范

1. 所有动效优先使用 `transform` 和 `opacity`，避免触发重排。
2. 使用 `will-change: transform` 提示浏览器优化，但不可滥用（仅用于正在动画的元素）。
3. 粒子背景独立 Canvas，不触发 React 重渲染。
4. 长列表动画使用虚拟滚动，仅动画可视区域元素。
5. 动画帧率目标：桌面 60fps，移动端 30fps。
6. 使用 `requestAnimationFrame` 而非 `setInterval` 驱动动画。

### 6.14 prefers-reduced-motion 支持

所有动效必须尊重用户的 `prefers-reduced-motion` 设置：

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

降级策略：
- 页面过渡：直接显示，无 fadeInUp。
- 卡片 stagger：全部同时显示，无错峰。
- 打字机效果：直接显示全文。
- 粒子背景：静止或极慢移动（速度降为 0.1）。
- 仪式动画：缩短为 0.5s 简化版。
- 冲突脉冲：改为静态红色连线，无脉冲。

## 7. 交互规范

### 7.1 议会流程交互

议会是 LifeVerse 的核心交互场景，涵盖 Wisdom Council（智慧议会）与 Future Council（未来议会）。完整流程包含 6 个阶段。

#### 7.1.1 议会流程总览

```
提问 → 选人 → 仪式 → 3 轮辩论 → 报告 → 时间线
```

#### 7.1.2 阶段一：提问

| 交互元素 | 规范 |
| --- | --- |
| 入口 | 首页「召开议会」Gold 按钮 / 导航栏议会入口 |
| 议题输入 | 居中大字号输入框（H3 字号），placeholder「你想探讨什么？」 |
| 议题分类 | 6 个分类标签（职业/关系/健康/财务/成长/其他），单选 |
| 背景信息 | 可选多行输入框，placeholder「补充一些背景...」 |
| 字数限制 | 议题 500 字，背景 2000 字 |
| 提交按钮 | Primary 按钮「开启议会」，议题为空时禁用 |
| 智能建议 | 输入议题后，AI 推荐 3 个相关历史议会（如有） |

交互细节：
- 议题输入框获得焦点时，金色光晕扩散，营造"被倾听"的感觉。
- 输入超过 100 字时，显示字数计数器（次要色）。
- 分类标签 hover 时显示该分类的说明 tooltip。

#### 7.1.3 阶段二：选人

| 议会类型 | 选人方式 | 默认成员 |
| --- | --- | --- |
| 智慧议会 | 系统推荐 3-7 位，用户可增删 | 7 位智者全部出席 |
| 未来议会 | 固定 4 位（20/当前/50/80 岁自己） | 不可更改 |
| 扩大议会 | 智慧议会 + 未来议会联合 | 11 位成员 |

交互细节：
- Agent 卡片网格排列，每张卡片可点击切换"出席/缺席"。
- 出席状态：卡片正常显示 + 金色边框。
- 缺席状态：卡片降低透明度（opacity 0.4）+ 灰度滤镜。
- 最少 3 位，最多 7 位（智慧议会），超出时提示。
- 底部显示已选人数「已选择 5/7」。

#### 7.1.4 阶段三：仪式

| 时间 | 动效 | UI 状态 |
| --- | --- | --- |
| 0-0.5s | 粒子汇聚 | 议题区淡出 |
| 0.3-2.5s | Agent 依次入场 | Agent 卡片从中心飞向环形位置 |
| 2.5-3.0s | 光柱 + 议题显现 | 中央显示议题，仪式完成 |

交互细节：
- 仪式期间不可操作，显示「议会正在召集...」文案。
- 仪式完成后，自动进入第一轮辩论。
- 提供「跳过仪式」按钮（Ghost 样式），尊重用户时间。

#### 7.1.5 阶段四：3 轮辩论

| 轮次 | 名称 | 内容 | 交互 |
| --- | --- | --- | --- |
| R1 | 各抒己见 | 每位 Agent 依次发言，给出初始立场 | 打字机逐字显示，可跳过 |
| R2 | 交叉质询 | Agent 之间互相提问与反驳 | 冲突连线可视化，用户可追加问题 |
| R3 | 共识凝聚 | 寻找整合方案，解决冲突 | 冲突逐步化解，共识标记出现 |

交互细节：
- 顶部进度条显示当前轮次（R1 → R2 → R3），分段式进度条。
- 当前发言 Agent 卡片高亮（scale 1.05 + 光晕脉冲）。
- 对话流底部固定用户输入框，R2/R3 阶段允许用户追加问题。
- 冲突发生时，两个冲突 Agent 之间出现红色脉冲连线。
- 共识达成时，相关 Agent 之间出现金色连线，持续 0.5s 后淡出。
- 每条发言显示 Agent 头像、名字、轮次标签、时间戳。
- 发言内容支持点赞/反对，反馈给 AI 微调权重。

#### 7.1.6 阶段五：报告

| 报告区块 | 内容 | 交互 |
| --- | --- | --- |
| 议题回顾 | 议题标题 + 分类 + 出席者 | 静态展示 |
| 各方立场 | 每位 Agent 的核心观点摘要 | 可展开查看完整发言 |
| 冲突值 | 数值 + 等级标签（微光/涟漪/风暴/裂痕） | 冲突值动画跳动 |
| 价值雷达漂移 | 决策前后雷达对比图 | compare 模式雷达图 |
| 共识方案 | 整合方案文字 | 打字机效果呈现 |
| 最终建议 | 留给用户思考的问题 | 金色强调，仪式感收束 |

交互细节：
- 报告生成动画：光晕汇聚 → 议题区旋转 → 报告展开。
- 报告支持导出为图片（html2canvas）。
- 报告底部固定操作栏：保存到时间线 / 重新召开 / 分享。
- 报告结尾永远以"最终的选择权在你"收束。

#### 7.1.7 阶段六：时间线

- 报告保存后自动写入 History 时间线。
- 时间线以卡片形式展示，显示议题、日期、冲突值、出席者头像。
- 点击时间线卡片可回看完整议会记录（只读模式）。
- 回看模式下可"续议"——基于历史议会继续讨论。

记忆星球是 Memory Planet 模块的核心交互，涵盖记忆的完整生命周期。

#### 7.2.1 流程总览

| 拖拽上传 | 拖拽文件到上传区域 → 区域高亮 → 释放上传 |

交互细节：
- 上传区域为虚线边框 + 金色 dashed，hover 时边框变实线金色。
- 拖拽时上传区域背景变为 `bg.gold-soft`。
- 上传进度条使用金色渐变 + shimmer 效果。
- 上传完成后，卡片以 stagger 动画依次入场。
- 支持多文件同时上传，每个文件独立进度条。

#### 7.2.3 AI 分类

- 上传后自动进入 AI 分类状态，卡片显示加载动画（三点动画）。
- 分类完成后，卡片显示所属星球标签（5 色之一）。
- 低置信度分类时，弹出友好询问：「这条记忆似乎同时属于'青春森林'和'爱情海洋'，你希望放在哪里？」
- 用户可随时手动修正分类，修正后卡片飞向新星球（过渡动画）。

#### 7.2.4 浏览

| 视图层级 | 交互 |
| --- | --- |
| 宇宙视图 | 5 个星球环绕排列，可旋转、缩放 |
| 星系视图 | 单个星球的地形展开 |
| 区域视图 | 某类记忆的集合 |
| 记忆视图 | 单条记忆详情 |

交互细节：
- 宇宙视图支持鼠标拖拽旋转、滚轮缩放。
- 每个星球有专属颜色光晕（青春森林绿、爱情海洋蓝、家庭小镇金、梦想之城紫、成长山脉白）。

- 顶部时间轴滑块，拖动可查看不同年份的记忆分布。
- 左侧筛选栏：按人物、主题、情感色调、时间筛选。
- 点击星球进入星系视图，点击记忆进入详情。

#### 7.2.5 详情

- 记忆详情以全屏弹窗呈现，毛玻璃背景。
- 展示原始文件（照片/视频/文字/音频）。
- 展示元数据：时间、地点、人物、情感色调、重要度。
- 展示 AI 生成的记忆描述（打字机效果）。
- 底部操作栏：编辑 / 回放 / 对话 / 删除 / 修改隐私。

#### 7.2.6 回放

- 点击「回放」进入沉浸式回放模式，全屏黑色背景。
- 照片/视频以电影感呈现（缓慢缩放 Ken Burns 效果）。
- AI 语音讲述当时情境（基于元数据与用户补充）。
- 背景音乐与情感色调匹配（暖色=温暖音乐，冷色=清冷音乐）。
- 回放结束时，弹出「现在的你想对当时的自己说什么？」输入框。
- 用户输入的内容写入 History，成为"跨时空对话锚点"。

### 7.3 时间线推演交互

时间线推演是 Future Council 的核心交互，让用户在行动前看见未来的多种可能。

#### 7.3.1 流程总览

```
输入决策 → 3 条分支 → 4 个自己发言 → 后悔分析 → 报告
```

#### 7.3.2 输入决策

- 决策输入框与议会议题输入类似，但 placeholder 为「你正在考虑什么决定？」
- 提供决策分类：职业/关系/健康/财务/居住/其他。
- 可选附加条件：「假设我会在 3 年后结婚」等扰动条件。
- 系统根据决策生成 3 条时间线分支。

#### 7.3.3 3 条分支

| 分支 | 含义 | 视觉 |
| --- | --- | --- |
| 时间线 A | 选择 X | 金色路径 |
| 时间线 B | 选择 Y | 蓝色路径 |
| 时间线 C | 不选择 | 灰色路径 |

交互细节：
- 3 条分支以 SVG 路径图呈现，从当前时间点向右延伸。
- 每条路径上有 3 个快照节点：1 年后、5 年后、10 年后。
- 点击节点展开快照详情（职业/关系/健康/心理/价值雷达）。
- 路径绘制使用 `stroke-dashoffset` 动画，依次绘制。
- 每条路径附带置信度标签（1 年 70%、5 年 40%、10 年 20%）。

#### 7.3.4 4 个自己发言

| 成员 | 视角 | 发言风格 |
| --- | --- | --- |
| 20 岁自己 | 理想、热血 | 激进、理想化 |
| 当前自己 | 现实、平衡 | 理性、权衡 |
| 50 岁自己 | 家庭、事业 | 稳重、务实 |
| 80 岁自己 | 回望、传承 | 智慧、超然 |

交互细节：
- 4 个自己以时间轴排列（左→右：20 → 当前 → 50 → 80）。
- 依次发言，打字机效果，每人发言后停顿 1s。
- 发言内容基于其时间视角评价 3 条分支。
- 80 岁自己的发言包含后悔分析，以特殊样式（金色边框）标记。
- 用户可点击任意"自己"查看其完整价值雷达。

#### 7.3.5 后悔分析

- 后悔分析是时间线推演的核心交付物，由 80 岁自己"回望"给出。
- 展示每种选择的后悔概率（接受 38% / 拒绝 52%）。
- 后悔概率以环形进度条呈现，≥ 50% 时红色警示。
- 标注主要后悔类型（行动后悔/不行动后悔/速度后悔/方向后悔/关系后悔）。
- 80 岁自己的发言以引号包裹的文学化形式呈现，不打分不评判。
- 后悔分析永远以"关键问题不是哪个不后悔，而是你更愿意承担哪种后悔"收束。

#### 7.3.6 报告

- 未来推演报告结构与智慧议会报告类似，但增加时间线推演专属区块。
- 包含：决策回顾、3 条时间线对比、4 个自己的立场、后悔分析、时间冲突值。
- 时间冲突值以等级标签呈现（时空一致/涟漪/撕裂/裂痕）。
- 冲突值 ≥ 0.5（撕裂）时，建议联合 Wisdom Council 召开扩大议会。
- 报告保存到 History，成为后续推演的"先例"。

### 7.4 错误处理交互

LifeVerse 的错误处理遵循"温柔而非冷酷"原则，所有错误提示必须有温度。

#### 7.4.1 错误类型与处理

| 错误类型 | 提示文案 | 交互 |
| --- | --- | --- |
| 网络错误 | 「宇宙信号不稳定，请稍后再试」 | 重试按钮 |
| AI 生成超时 | 「智者们还在沉思，再等一会儿？」 | 重试 / 跳过 |
| 议会生成失败 | 「这次议会未能完成，你的议题已保存」 | 重新召开 |
| 上传失败 | 「这条记忆没能送达，再试一次？」 | 重试 |
| 权限不足 | 「这个区域需要先完成自我画像」 | 引导跳转 |
| 数据为空 | 「你的宇宙还在等待第一颗星辰」 | 引导 CTA |

#### 7.4.2 错误提示规范

- 错误提示使用 Toast 形式，从顶部滑入，3s 后自动消失。
- Toast 背景为 `bg.card` + 警告色左边框（4px）。
- 错误图标 + 文案 + 操作按钮（如有）。
- 禁止使用「Error」「失败」「出错了」等冰冷词汇。
- 禁止使用 alert/confirm 弹窗，全部使用自定义组件。
- 严重错误（数据丢失风险）使用弹窗，需用户确认。

#### 7.4.3 重试机制

- 所有网络请求失败后提供重试按钮。
- 重试按钮使用 Secondary 样式，文案「再试一次」。
- 重试次数超过 3 次后，提示「似乎网络有问题，稍后再来看看？」
- AI 生成类请求，重试时使用相同参数，避免重复计费。
- 议会类长任务，支持断点续传（已生成的发言保留）。

### 7.5 空状态交互

空状态是用户首次接触某功能的时刻，必须兼具引导与温度。

#### 7.5.1 空状态设计原则

## 8. 响应式规范

### 8.1 断点系统

LifeVerse 采用移动优先的响应式策略，使用 TailwindCSS 的标准断点系统。

| 断点 | 宽度阈值 | Tailwind 前缀 | 目标设备 |
| --- | --- | --- | --- |
| 默认 | < 640px | （无前缀） | 手机竖屏 |
| sm | ≥ 640px | `sm:` | 手机横屏 / 小平板 |
| md | ≥ 768px | `md:` | 平板竖屏 |
| lg | ≥ 1024px | `lg:` | 平板横屏 / 小桌面 |
| xl | ≥ 1280px | `xl:` | 桌面 |
| 2xl | ≥ 1536px | `2xl:` | 大桌面 |

```typescript
// tailwind.config.ts 断点（使用 Tailwind 默认值，无需修改）
screens: {
  sm: '640px',
  md: '768px',
  lg: '1024px',
  xl: '1280px',
  '2xl': '1536px',
}
```

### 8.2 移动端导航

移动端（< 768px）导航采用汉堡菜单方案：

| 元素 | 移动端表现 |
| --- | --- |
| Logo | 左侧，缩小为 20px 字号 |
| 汉堡按钮 | 右侧，24px 图标 |
| 主导航 | 隐藏，点击汉堡展开全屏菜单 |
| 议会入口 | 移入汉堡菜单顶部 |
| 用户头像 | 移入汉堡菜单底部 |

汉堡菜单交互：
- 点击汉堡按钮，菜单从右侧滑入（x: 100% → 0，duration 0.3s）。
- 菜单全屏覆盖（100vw × 100vh），背景为 `bg.card` + 强毛玻璃。
- 菜单项垂直排列，每项 56px 高，字号 18px，左侧金色图标。
- 当前页菜单项金色高亮。
- 点击菜单项或关闭按钮，菜单滑出（x: 0 → 100%，duration 0.3s）。
- 菜单打开时禁止背景滚动。

### 8.3 卡片网格响应式

卡片网格根据屏幕宽度自适应列数：

| 断点 | 列数 | 间距 | Tailwind 类 |
| --- | --- | --- | --- |
| < 640px（移动） | 1 列 | 16px | `grid grid-cols-1 gap-4` |
| 640-1024px（平板） | 2 列 | 24px | `grid grid-cols-2 gap-6` |
| ≥ 1024px（桌面） | 3 列 | 24px | `grid grid-cols-3 gap-6` |
| ≥ 1280px（大桌面） | 3-4 列 | 32px | `grid grid-cols-3 xl:grid-cols-4 gap-8` |

特殊网格：
- 议会大厅 Agent 卡片：移动端 2 列网格，桌面端环形布局。
- 记忆星球：移动端列表视图，桌面端宇宙视图。
- 时间线：移动端单列纵向，桌面端双列交错。

### 8.4 字体缩放

字体使用 `clamp()` 实现流体响应式，避免断点处字号跳变。

```css
/* H1: 48px → 80px */
font-size: clamp(48px, 5vw, 80px);

/* H2: 32px → 48px */
font-size: clamp(32px, 4vw, 48px);

/* H3: 20px → 24px */
font-size: clamp(20px, 2vw, 24px);

/* H4: 18px → 20px */
font-size: clamp(18px, 1.5vw, 20px);

/* Body: 固定 16px（不缩放，保证可读性） */
font-size: 16px;
```

Tailwind 响应式字号映射：

```typescript
fontSize: {
  h1: ['clamp(48px, 5vw, 80px)', { lineHeight: '1.1' }],
  h2: ['clamp(32px, 4vw, 48px)', { lineHeight: '1.2' }],
  h3: ['clamp(20px, 2vw, 24px)', { lineHeight: '1.3' }],
  h4: ['clamp(18px, 1.5vw, 20px)', { lineHeight: '1.4' }],
}
```

### 8.5 议会大厅响应式

议会大厅是布局最复杂的页面，需要特殊响应式处理。

| 断点 | Agent 布局 | 环形半径 | 对话流高度 | 中央议题区 |
| --- | --- | --- | --- | --- |
| < 640px | 水平滚动列表 | N/A | 50vh | 隐藏，议题显示在顶部 |
| 640-1024px | 环形（small 尺寸） | 200px | 40vh | 160px 直径 |
| 1024-1440px | 环形（medium 尺寸） | 280px | 40vh | 200px 直径 |
| ≥ 1440px | 环形（large 尺寸） | 320px | 35vh | 280px 直径 |

移动端议会大厅特殊处理：
- Agent 卡片改为水平滚动列表，每屏显示 2.5 张卡片。
- 对话流移至全屏，Agent 列表收起为顶部横条。
- 冲突连线在移动端改为冲突标记图标（Agent 卡片角标）。
- 仪式动画简化为 Agent 卡片依次淡入。

### 8.6 触摸交互优化

移动端触摸交互与桌面端鼠标交互有本质差异，需要专门优化：

| 交互 | 桌面端 | 移动端 |
| --- | --- | --- |
| hover | 鼠标悬停触发 | 移除 hover 效果，改为 active 状态 |
| 点击 | onClick | onTouchEnd（消除 300ms 延迟） |
| 滚动 | 滚轮 | 触摸滑动，惯性滚动 |
| 缩放 | Ctrl + 滚轮 | 双指捏合 |
| 右键菜单 | 右键点击 | 长按 |
| 拖拽 | 鼠标拖拽 | 长按后拖拽 |

触摸优化规范：
- 所有可点击元素最小触摸区域 44×44px（Apple HIG 标准）。
- 移动端移除所有 hover 依赖的交互，改为 tap 或 long-press。
- 滚动区域添加 `-webkit-overflow-scrolling: touch` 实现惯性滚动。
- 禁用移动端的文字选择（除正文内容外），避免误触。
- 添加 `touch-action: manipulation` 消除双击缩放。

### 8.7 响应式图片

```typescript
// 使用 next/image 响应式图片
import Image from 'next/image';

<Image
  src="/hero.jpg"
  alt="LifeVerse"
  width={1920}
  height={1080}
  sizes="(max-width: 640px) 100vw, (max-width: 1024px) 50vw, 33vw"
  priority
/>
```

图片响应式规范：
- 使用 `next/image` 自动优化格式（WebP/AVIF）与尺寸。
- `sizes` 属性必须设置，帮助浏览器选择合适尺寸。
- 首屏图片添加 `priority` 属性预加载。
- 头像使用固定尺寸（32/48/64px），非首屏使用 `loading="lazy"`。
- 背景图使用 CSS `background-size: cover` + `picture` 元素适配。

### 8.8 安全区适配

移动端需要适配刘海屏、底部安全区：

```css
/* 顶部安全区 */
padding-top: env(safe-area-inset-top);

/* 底部安全区 */
padding-bottom: env(safe-area-inset-bottom);

/* 左右安全区（横屏） */
padding-left: env(safe-area-inset-left);
padding-right: env(safe-area-inset-right);
```

- 固定吸顶元素（导航栏）顶部添加 `safe-area-inset-top`。
- 固定吸底元素（底部操作栏）底部添加 `safe-area-inset-bottom`。
- 设置 `viewport-fit=cover` 启用安全区。

## 9. 无障碍规范

### 9.1 无障碍设计原则

LifeVerse 致力于让每个人都能使用，包括视障、听障、运动障碍与认知障碍用户。无障碍不是可选的附加项，而是设计的一等公民。我们遵循 WCAG 2.1 AA 级别标准，部分指标达到 AAA 级别。

### 9.2 颜色对比度

| 文字类型 | 色值组合 | 对比度 | 标准 | 达标 |
| --- | --- | --- | --- | --- |
| 主文字 | `#e8e6e3` on `#060710` | 15.8:1 | ≥ 4.5:1（AA） | AAA |
| 次要文字 | `#8a8892` on `#060710` | 6.9:1 | ≥ 4.5:1（AA） | AA |
| 金色文字 | `#c9a84c` on `#060710` | 7.2:1 | ≥ 4.5:1（AA） | AAA |
| 成功色 | `#4ade80` on `#060710` | 9.1:1 | ≥ 4.5:1（AA） | AAA |
| 警告色 | `#f87171` on `#060710` | 5.8:1 | ≥ 4.5:1（AA） | AA |
| 信息色 | `#60a5fa` on `#060710` | 6.3:1 | ≥ 4.5:1（AA） | AA |
| 微弱文字 | `#555560` on `#060710` | 2.8:1 | ≥ 3:1（大文字） | 仅限装饰 |

对比度规则：
- 正文文字对比度必须 ≥ 4.5:1。
- 大文字（≥ 24px 或 ≥ 18.66px bold）对比度必须 ≥ 3:1。
- 微弱文字（`#555560`）对比度仅 2.8:1，仅用于非关键装饰性文字，不可承载语义信息。
- 交互元素边框对比度必须 ≥ 3:1。
- 焦点指示器对比度必须 ≥ 3:1。

### 9.3 键盘导航

所有交互元素必须可通过键盘完整操作。

#### 9.3.1 键盘操作规范

| 按键 | 行为 |
| --- | --- |
| Tab | 向前聚焦下一个可交互元素 |
| Shift + Tab | 向后聚焦上一个可交互元素 |
| Enter | 触发当前聚焦元素（按钮、链接） |
| Space | 触发当前聚焦元素（按钮、复选框） |
| ESC | 关闭弹窗 / 取消操作 / 退出编辑 |
| 方向键 | 在列表、标签组、网格中移动 |
| Home / End | 跳到列表首项 / 末项 |

#### 9.3.2 Tab 顺序

- Tab 顺序必须与视觉阅读顺序一致（从左到右，从上到下）。
- 隐藏元素（`display: none` / `visibility: hidden`）不参与 Tab 顺序。
- 弹窗打开时，Tab 焦点陷阱在弹窗内循环。
- 弹窗关闭后，焦点返回触发元素。

#### 9.3.3 跳过导航

每个页面提供"跳过导航到主内容"链接，供键盘用户快速跳过重复的导航栏。

```html
<a href="#main-content" class="skip-link">跳过导航</a>
<!-- 导航栏 -->
<nav>...</nav>
<!-- 主内容 -->
<main id="main-content">...</main>
```

```css
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #c9a84c;
  color: #060710;
  padding: 8px 16px;
  z-index: 100;
}
.skip-link:focus {
  top: 0;
}
```

### 9.4 ARIA 标签

#### 9.4.1 ARIA 使用原则

- 优先使用语义化 HTML（`<nav>`、`<main>`、`<button>`、`<a>`），而非 ARIA 角色。
- ARIA 仅在语义化 HTML 不足时使用。
- 永不使用 ARIA 覆盖原生 HTML 语义。
- 所有 ARIA 标签必须使用中文（与界面语言一致）。

#### 9.4.2 常用 ARIA 规范

| 场景 | ARIA 属性 | 示例 |
| --- | --- | --- |
| 导航 | `role="navigation" aria-label="主导航"` | `<nav aria-label="主导航">` |
| 主内容 | `role="main"` | `<main>` |
| 按钮 | `aria-label="开启议会"` | `<button aria-label="开启议会">` |
| 弹窗 | `role="dialog" aria-modal="true" aria-labelledby="title"` | `<dialog>` |
| 加载状态 | `role="status" aria-live="polite"` | `<div aria-live="polite">加载中</div>` |
| 错误提示 | `role="alert" aria-live="assertive"` | `<div role="alert">网络错误</div>` |
| 标签页 | `role="tablist"` / `role="tab"` / `aria-selected` | 标签页组件 |
| 雷达图 | `aria-label="价值雷达图：自由80，财富60..."` | SVG 图表 |
| 粒子背景 | `aria-hidden="true"` | 装饰性 Canvas |
| 打字机 | `aria-label={完整文本}` | 逐字显示组件 |

#### 9.4.3 动态内容播报

| 场景 | aria-live | 说明 |
| --- | --- | --- |
| 议会新发言 | `polite` | 不打断当前操作，空闲时播报 |
| 错误提示 | `assertive` | 立即播报 |
| 加载完成 | `polite` | 空闲时播报 |
| 冲突发生 | `polite` | 空闲时播报 |
| 进度更新 | `off` | 不播报（避免频繁打断） |

### 9.5 focus-visible 样式

所有可交互元素必须有清晰的 `focus-visible` 样式，且仅在使用键盘聚焦时显示（鼠标点击不显示）。

```css
/* 全局 focus-visible 样式 */
*:focus-visible {
  outline: 2px solid #c9a84c;              /* 金色描边 */
  outline-offset: 2px;                      /* 描边偏移 */
  box-shadow: 0 0 0 4px rgba(201, 168, 76, 0.2); /* 金色光晕 */
}

/* 移除鼠标点击的 focus 样式 */
*:focus:not(:focus-visible) {
  outline: none;
  box-shadow: none;
}
```

focus-visible 规范：
- 描边颜色为金色 `#c9a84c`，与品牌色一致。
- 描边宽度 2px，偏移 2px，确保不遮挡元素内容。
- 配合金色光晕（4px 扩散），提升可见性。
- 永不使用 `outline: none` 移除所有焦点样式。

### 9.6 prefers-reduced-motion 支持

尊重用户系统的减少动效偏好：

```css
@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

各动效的降级策略见 6.14 节。

#### 9.7.1 语义化结构

```html

  <nav aria-label="主导航">...</nav>
</header>
<main role="main" id="main-content">
  <h1>页面标题</h1>

    <h2 id="section-title">区块标题</h2>
    ...
  </section>

```

#### 9.7.2 图片替代文本

| 图片类型 | alt 文本 | 示例 |
| --- | --- | --- |
| 内容图片 | 描述图片内容 | `alt="大学毕业典礼合影"` |
| 装饰图片 | 空 alt | `alt=""` |
| 功能图标 | 描述功能 | `alt="关闭"` |

#### 9.7.3 表单无障碍

```html
<label for="topic">议会议题</label>
<input
  id="topic"
  type="text"
  aria-required="true"
  aria-invalid="false"
  aria-describedby="topic-help topic-error"
/>
<span id="topic-help">请输入你想探讨的问题</span>
<span id="topic-error" role="alert">议题不能为空</span>
```

表单规范：
- 所有输入框必须有关联的 `<label>`。
- 必填字段使用 `aria-required="true"`。
- 校验错误使用 `aria-invalid="true"` + `role="alert"`。
- 帮助文字使用 `aria-describedby` 关联。

### 9.8 认知无障碍

| 规范 | 说明 |
| --- | --- |
| 一致性 | 相同功能使用相同交互模式，避免用户重新学习 |
| 可预测性 | 操作结果可预测，无意外跳转或弹窗 |
| 容错性 | 破坏性操作（删除、导出）需二次确认 |
| 可撤销 | 尽可能提供撤销操作（如删除后 5s 内可撤销） |
| 清晰反馈 | 每个操作都有明确的视觉/文字反馈 |
| 简洁文案 | 避免长句，使用短句与列表，降低认知负荷 |
