# LifeVerse MVP 范围定义

> 文档版本：v1.0
> 维护者：产品总监 Alex Chen、市场总监 Rachel Bai
> 上游文档：`prd-v5.md`、`user_story.md`、`user_flow.md`
> Phase：Phase 1 MVP

---

## 1. MVP 目标

### 1.1 一句话目标

让 1000 名种子用户在 LifeVerse 中召开自己的第一次智慧议会与未来议会，并收到一份改变其决策视角的命运报告。

### 1.2 MVP 北极星

**首次议会完成率**：注册后 7 天内至少召开 1 次议会（智慧或未来）的用户比例，目标 ≥ 60%。

### 1.3 MVP 验证假设

| 假设 | 验证方式 | 成功标准 |
| --- | --- | --- |
| 用户愿意把人生决策交给"7 位智者"审议 | 首次议会完成率 | ≥ 60% |
| 多元视角确实能帮用户看清决策 | 报告满意度评分 | ≥ 4.0/5 |
| 用户愿意为"未来推演"持续回访 | 7 日留存 | ≥ 35% |
| 价值雷达能引发自我觉察 | 雷达漂移查看率 | ≥ 50% |

## 3. MVP 不包含范围

### 3.1 明确不包含的模块

- **Inner World**：6 人格、内心天气、内心议会 → Phase 2
- **Memory Planet**：记忆上传、5 星球、人生地图 → Phase 2
- **Dream Archive**：梦想记录、儿时自己 → Phase 3
- **Reunion**：AI 亲人、私人议会 → Phase 3

### 3.2 明确不包含的功能

| 功能 | 不包含原因 | 计划 Phase |
| --- | --- | --- |
| 记忆上传与分类 | 依赖多模态管线，MVP 聚焦决策 | Phase 2 |
| 内心人格对话 | 需要情绪模型积累 | Phase 2 |
| AI 亲人生成 | 伦理风险高，需独立审计 | Phase 3 |
| 儿时自己 AI | 依赖童年记忆数据 | Phase 3 |
| 生命星图 | 需要足够多的事件数据 | Phase 4 |
| 年度生命报告 | 需要一年数据积累 | Phase 4 |
| 协作议会 | 需要权限模型完善 | Phase 4 |
| 开放 API | 需要稳定的核心 API | Phase 4 |
| 多语言 | MVP 仅中文 | Phase 2 |
| 移动端原生 | MVP 为 Web + PWA | Phase 3 |

### 3.3 明确不包含的非功能

- 99.9% SLA（MVP 目标 99.5%）
- 离线模式
- 端侧 AI 推理
- 自定义智者（MVP 仅 7 位固定智者）

## 4. MVP 验收标准

### 4.1 功能验收

| 验收项 | 标准 | 验证方式 |
| --- | --- | --- |
| 注册与自我画像 | 5 分钟内完成，数据落库 | E2E 测试 |
| 智慧议会 | 7 智者发言 + 2 轮质询 + 报告生成 < 30s | 集成测试 |
| 价值雷达 | 5 维可视化 + 漂移计算正确 | 单元测试 |
| 未来议会 | 4 时间自己 + 3 时间线 + 后悔分析 | 集成测试 |
| History | 时间线渲染 < 2s，事件可跳转 | E2E 测试 |
| 扩大议会 | WC + FC 联合报告生成 | 集成测试 |

### 4.2 性能验收

| 指标 | 目标 | 测量方式 |
| --- | --- | --- |
| 首屏 LCP | < 2.0s | Lighthouse |
| 议会首 token | < 1.5s | 后端日志 |
| 议会完整生成 | < 30s | 后端日志 |
| 时间线渲染 | < 2s | E2E 性能 |
| 并发支持 | 500 并发用户 | 压测 |

### 4.3 质量验收

| 指标 | 目标 |
| --- | --- |
| 单元测试覆盖率 | ≥ 70% |
| E2E 测试覆盖核心流程 | 100% |
| P0 Bug 数 | 0 |
| P1 Bug 数 | ≤ 5 |
| Sentry 错误率 | < 1% |

### 4.4 安全验收

| 项 | 标准 |
| --- | --- |
| RLS 行级权限 | 所有表启用，通过渗透测试 |
| 数据加密 | 传输 TLS 1.3，存储 AES-256 |
| AI 数据隔离 | 用户数据不进入公共训练 |
| 心理危机干预 | 关键词检测 + 热线展示 |

### 4.5 用户体验验收

| 指标 | 目标 | 测量方式 |
| --- | --- | --- |
| 首次议会完成率 | ≥ 60% | 埋点 |
| 报告满意度 | ≥ 4.0/5 | 报告末尾评分 |
| 7 日留存 | ≥ 35% | 埋点 |
| NPS | ≥ 30 | 内测问卷 |

## 6. 技术债务

MVP 为追求速度，会主动承担以下技术债务，并在后续 Phase 偿还。

### 6.1 主动承担的债务

| 编号 | 债务 | 影响 | 偿还计划 |
| --- | --- | --- | --- |
| TD-001 | 智者 prompt 硬编码 | 难以扩展智者 | Phase 2 抽象为配置 |
| TD-002 | 议会报告无版本化 | 报告模板修改无法回溯 | Phase 2 引入版本 |
| TD-003 | 无缓存层 | LLM 成本高 | Phase 2 引入 Redis |
| TD-004 | 单 LLM 后端 | OpenAI 故障即停服 | Phase 2 多后端 |
| TD-005 | 无异步任务队列 | 长任务阻塞请求 | Phase 2 引入队列 |
| TD-006 | History 仅列表视图 | 缺少星图等可视化 | Phase 4 生命星图 |
| TD-007 | 无 i18n | 仅中文 | Phase 2 引入 i18n |
| TD-008 | 无移动端适配 | 仅 Web | Phase 3 PWA |
| TD-009 | 无 A/B 测试框架 | 难以实验 | Phase 2 引入 |
| TD-010 | 无数据仓库 | 难以做用户分析 | Phase 3 引入 |

### 6.2 债务管理原则

1. **显性化**：所有债务记录在 `tech_debt.md`，不在代码中隐藏。
2. **限时偿还**：每条债务标注偿还 Phase，不允许无限期拖延。
3. **不传染**：MVP 的债务不得影响 Phase 2 的架构决策，必要时先偿还再扩展。
4. **可量化**：每条债务标注"利息"——即不偿还每月带来的额外成本。

## 7. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
| --- | --- | --- | --- |
| LLM 成本超预算 | 中 | 高 | 缓存 + 降级 + 限流 |
| 智者发言质量不稳定 | 中 | 高 | Few-shot + 人工评审 + 持续优化 prompt |
| 用户对 AI 决策不信任 | 中 | 高 | 显性化冲突值 + 强调"镜子而非裁判" |
| 心理危机事件 | 低 | 极高 | 关键词检测 + 热线 + 法务预案 |
| 7 日留存不达标 | 中 | 高 | 内测期快速迭代引导流程 |

---

## 8. 成功标准汇总

MVP 视为成功的条件（全部满足）：

1. 1000 种子用户完成注册。
2. 首次议会完成率 ≥ 60%。
3. 7 日留存 ≥ 35%。
4. 报告满意度 ≥ 4.0/5。
5. P0 Bug = 0，P1 Bug ≤ 5。
6. 无心理危机负面事件。
7. 单用户 LLM 成本 ≤ 5 元/月。

满足以上条件后，启动 Phase 2 开发。
